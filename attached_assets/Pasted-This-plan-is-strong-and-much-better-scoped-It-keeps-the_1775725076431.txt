#37 — Task #37: AI Source Field Reuse — Explicit Justification Before New Field Creation

Problem

In the SOP upload flow, AI sometimes invents new business or derived fields instead of reusing an existing source field that may already capture the same concept. This happens because the prompt currently encourages reuse but does not force an explicit reuse-vs-create decision.

This leads to:

duplicate fields
unnecessary business/derived fields
reduced trust in generated output
harder policy review

Important: we are not solving this with fuzzy matching or backend similarity scoring. We want the AI to make a deliberate decision first, and explain itself when it creates something new.

Goal

Force AI to explicitly decide whether to:

reuse an existing source field, or
create a new business field, or
create a new derived field

If AI creates a new field, it must provide a creation_reason explaining why no existing source field was a reasonable fit.

We also want to:

store this reason
show it in UI
preserve existing backend field resolution
avoid building a fuzzy-matching engine
Not in scope

Do not build:

fuzzy matching
vector search
embeddings
similarity scoring engine
backend semantic resolution logic

Everything should remain:

prompt-driven
schema-validated
stored in existing JSONB draft objects
visible in review UI
Implementation
1) Prompt update — add FIELD REUSE DECISION section

File: server/ai-engine.ts

Add a new instruction block in the IMPORTANT RULES section.

Text to add:

FIELD REUSE DECISION — apply this checklist BEFORE creating any new business or derived field:

STEP 1: Check whether any existing source field in the Beacon field catalog already captures this concept.
Use the field label, description, and sample values (if provided) to judge fit.

STEP 2: If an existing source field is a reasonable semantic match, reuse that source field directly in the rule.
Use the exact field label from the catalog.
Do NOT create a new business or derived field just because a cleaner or more specific label sounds better.

STEP 3: Do NOT create wrapper fields around source fields.
If a new field would merely rename, restate, or lightly reformulate an existing source field, reuse the source field directly instead.

STEP 4: Only create a new business or derived field if no existing source field is a reasonable fit.
If you create a new field, you MUST include a creation_reason explaining why no existing source field was sufficient.
Be specific and name the closest candidate(s) when relevant.

Also add examples of good creation_reason values.

Important: do not imply that active_dca_placement and is_on_dmp are automatically equivalent. Use examples that demonstrate reasoning, not forced equivalence.
