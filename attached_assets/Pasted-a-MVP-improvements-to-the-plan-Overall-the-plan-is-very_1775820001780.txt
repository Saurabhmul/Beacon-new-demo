a) MVP improvements to the plan

Overall the plan is very good. For MVP I would improve these things:

1) Add treatment priority to the deterministic layer

This is the biggest missing piece.

Today the pipeline determines:

eligible treatments
blocked treatments

It should also determine:

ranked eligible treatments

At minimum:

sort by priority
lower number = higher priority
if no priority exists, treat as lowest priority or default rank

Why:
This reduces model freedom and makes decisions more repeatable.

What to tell Replit

Add a deterministic ranking step after rule evaluation:

take eligibleTreatments
sort by treatment priority ascending (1 = highest)
include ranked list in RuleEvaluationResult
pass ranked eligible treatments into final decision prompt

Also add:

rankedEligibleTreatments: Array<{
  code: string;
  name: string;
  priority: number | null;
  reasons: string[];
}>
2) Narrow final AI choice when priority is decisive

For MVP, if one eligible treatment has strictly أعلى priority than all others, the model should usually not “choose creatively.”

Why:
If policy has already said treatment A outranks B, AI should mostly explain, not re-decide.

What to tell Replit

Add this behavior in Task #40:

if exactly one highest-priority treatment exists, pass it as preferred_treatment
final AI should either:
recommend that treatment
or choose AGENT_REVIEW
or NO_ACTION
it should not pick a lower-priority eligible treatment unless explicit review/guardrail reasoning justifies it

This is a very strong MVP improvement.

3) Separate “eligibility” from “selection”

Right now these are blended a little.

You want 3 layers:

eligibility — which treatments are possible
ranking — among possible treatments, which is preferred by policy
selection/explanation — AI explains and outputs the final recommendation

Why:
This makes Beacon more stable across clients.

4) Add tie-handling for same priority

If two treatments are eligible and both have the same priority, Beacon needs a rule.

Suggested MVP rule

If multiple eligible treatments share the best priority:

pass all tied treatments to AI
AI chooses among only those tied treatments
if ambiguity remains, prefer AGENT_REVIEW

Why:
Simple and safe.

5) Add “priority missing” fallback

Some clients may not define priority cleanly yet.

Suggested MVP rule

If treatment priority missing:

treat as lowest precedence by default
but keep treatment eligible
log a warning in rule evaluation trace

Why:
Do not break the system because config is incomplete.

6) Add treatment selection trace

For MVP, don’t just store eligible vs blocked.

Also store:

priority
selection rank
why selected / not selected

Why:
This will help you debug “why did Beacon choose Payment Plan instead of Interest Freeze?”

7) Add one more validator rule in Task #40

Validator should check:

if recommended treatment is eligible but not highest-ranked,
then there must be a traceable reason:
review trigger
guardrail
missing info
explicit case-specific rationale

Why:
Otherwise lower-priority treatment choices become arbitrary.

8) Keep #41 lean for MVP

UI plan is good, but for MVP don’t over-invest in polish before backend stabilizes.

My only suggestion:

build the sections
make them readable
defer advanced polish if needed

The main value is in decision quality, not UI bells and whistles.

b) Deep dive on Task #40 and priority
Yes — priority should matter

You are right.

If multiple treatments are eligible, Beacon should not ask AI to pick freely from all of them without structure.

That defeats part of the reason you built:

SOP configuration
treatment rules
guardrails
structured policy

Priority exists exactly to solve:

“more than one treatment is applicable — which one should we prefer?”

So priority should be treated as a deterministic policy signal, not just descriptive metadata.

Where priority should enter the pipeline
Best place: after rule evaluation, before final AI

I strongly recommend this flow:

Step 1: Rule evaluator

Compute:

eligible treatments
blocked treatments
review triggers
guardrails
Step 2: Priority ranking layer

Take only eligible treatments and rank them by:

priority ascending (1 highest)
then maybe tie-break rules if needed
Step 3: Final AI

AI receives:

eligible treatments
ranked eligible treatments
preferred treatment(s)
blocked treatments
review triggers
guardrails
decision packet

Then AI’s job becomes:

explain customer situation
recommend the top-ranked eligible treatment
or choose AGENT_REVIEW / NO_ACTION if the case is unsafe or ambiguous

That is much better than letting AI choose from raw eligible treatments.
