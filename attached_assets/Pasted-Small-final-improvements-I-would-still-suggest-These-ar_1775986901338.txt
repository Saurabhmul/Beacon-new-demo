Small final improvements I would still suggest

These are not blockers, just polish.

1. Define what “safe mapping” means for evidence normalization

Right now it says:

normalize if safely mapped

I would ask Replit to define a tiny mapping table, for example:

source → source_field
payment_history → payment
conversation_note → conversation
bureau_record → bureau

Why:
otherwise “safe mapping” can become inconsistent.

2. Make confidence downgrade explicit after retry

If the first response fails validation and the second succeeds, I would recommend:

keep the field if valid
but do not allow extremely high confidence unless the evidence supports it

Why:
a repaired answer should not automatically look ultra-confident.

3. Clarify empty-object vs empty-array by section

The plan says empty arrays/objects when absent, which is good.

I would ask Replit to define which sections are arrays and which are objects.

For example:

customerProfile → object
resolvedSourceFields → object
loanData, paymentData, conversationData, bureauData, incomeEmploymentData → arrays or structured objects, but defined consistently
compliancePolicyInternalRules → array
knowledgeBaseAgentGuidance → array

Why:
stable prompt shape only works if section shapes are also stable.

4. Add one line on null rationale consistency

When a field becomes null due to:

timeout
schema failure
insufficient evidence

make sure null_reason is always set consistently and rationale is either empty or still concise and factual.

Why:
prevents messy traces.

5. Clarify where included_context_sections should come from

I would ask Replit to record a section as “included” only if it is non-empty after truncation/preprocessing, not just because the key exists.

Why:
otherwise every field trace may show all 10 sections even when most were empty placeholders.
