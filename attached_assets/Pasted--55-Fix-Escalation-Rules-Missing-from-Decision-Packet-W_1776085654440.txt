#55 - Fix Escalation Rules Missing from Decision Packet What & Why The AI decision engine never receives the escalation rules configured in the UI (including the vulnerability_rag != None → agent review rule). This means customers flagged as vulnerable are NOT being routed to human review — the AI just picks the best matching treatment and ignores all configured escalation conditions. Root cause: in server/routes.ts, the buildDecisionPacket() call that assembles the AI's input is missing three critical fields: compliancePolicyInternalRules — already computed as frozenComplianceRules (includes the vulnerability definition text) but never passed in. escalationRules — the structured escalation rules from policyConfig.escalationRules (includes the vulnerability_rag != None structured rule from the UI) but never passed in. bureauData and incomeEmploymentData — customer-level credit bureau and income/employment data are fetched and stored in data.bureau / data.income but also never passed in. Both frozenComplianceRules and policyConfig are already in scope at the call site — no new data fetching is needed. Done looks like Customers with vulnerability_rag != None (or any other configured escalation rule that fires) are correctly routed to agent review instead of receiving an auto-selected treatment. The == ESCALATION RULES == section in the AI prompt is no longer [] — it contains the actual configured escalation conditions. The == COMPLIANCE POLICY INTERNAL RULES == section is no longer [] — it contains the vulnerability definition text. Re-running analysis for the Lendable demo dataset routes the affected customers (CUST-003, CUST-004, CUST-005 etc.) to agent review when their vulnerability RAG is non-None. Out of scope Changes to the UI or escalation rule configuration screens. Changes to how escalation rules are stored in the database. Changes to the schema or the EscalationRules type.
