2) allowed_values should mainly matter for enum

Right now the plan says tag input for allowed values, shown when type is enum or always.

I would simplify:

always store allowed_values

That keeps UI cleaner.

3) default_value should stay as raw text in UI

For MVP, I would not over-parse default values in the form layer.

Better:

let user enter/edit raw default value
convert safely later at runtime based on resolved type

That avoids UI complexity.

4) Derived-field type deduction should be “best effort”

Good to deduce it, but do not present it as guaranteed truth.

I would ask Replit to label it internally as:

deduced type
editable by user

And if formula is mixed/unclear, fallback to string or null + warning.

5) Add one simple warning rule for bad formulas

You already say no blocking validation, only warn. Good.

I would make the warning explicit for cases like:

arithmetic formula using enum/string inputs
boolean formula using free text without coercion

Do not block, just warn in UI and at runtime return null when unsafe.

6) Runtime type priority should be made very explicit

This is one of the most important pieces.

Tell Replit to implement exactly this order:

user-defined type
AI-extracted type
system-deduced type
fallback string

And use this consistently everywhere.

7) Business field type inference should be simple

This line is good:

If data_type is missing, infer from allowed_values or description when possible, fallback to string.

I would tighten it slightly:

if allowed_values exists and has a small fixed set, prefer enum
if description clearly says yes/no or true/false, prefer boolean
if description clearly refers to amount/count/percentage, prefer number
else fallback to string

That is enough for MVP.

8) Safe coercion helpers should never silently invent values

Very important:

"251" → 251 is okay
"Yes" → true is okay
"HIGH" → number is not okay
invalid date string → null, not fake date

This should be strict.
